오늘은 패키지 레지스트리(특히 npm과 PyPI)의 인증·권한 강화와 관련해 공부한 내용을 정리하려고 합니다. 패키지 공급망 공격이 계속 이슈가 되고 있어서 개인 계정의 2FA 설정, CI용 토큰 정책, 토큰 회전과 최소 권한 원칙 등 점검 포인트를 중심으로 실무에서 확인하면 좋을 것들을 정리합니다. 저는 아직 배우는 입장이라 일부 내용은 문서와 실습을 통해 확인한 결과를 바탕으로 조심스럽게 정리합니다.

공부하면서 알게 된 점

- 왜 중요한가: 퍼블릭/프라이빗 레지스트리의 인증 정보가 유출되면 패키지 배포 권한을 탈취당해 악성 패키지로 배포될 위험이 있습니다. 특히 CI에 하드코딩된 토큰이나 오래된 퍼스널 액세스 토큰이 취약점이 될 수 있었어요.
- 계정 2FA(이중 인증)는 가능한 한 켜두는 것이 좋았습니다. 2FA가 있으면 단순 비밀번호 유출로는 배포 권한까지 넘어가는 것을 방지할 수 있는 여지가 큽니다.
- 토큰은 가능하면 프로젝트(패키지) 단위로, 역사적으로 회전(rotation) 주기를 짧게 두는 것이 안전하다고 느꼈습니다.
- CI에 토큰을 넣을 때는 시크릿 매니저(예: GitHub Actions Secrets, GitLab CI/CD variables, AWS Secrets Manager 등)를 사용하고, 로그에 노출되지 않도록 주의해야 합니다.

처음에는 헷갈렸던 부분

- npm과 PyPI의 토큰/2FA 모델이 조금 달라서 혼란스러웠습니다. 예를 들어 npm은 CLI를 통해 토큰을 만들고 관리하는 기능이 있고, PyPI는 웹 UI에서 프로젝트 범위 토큰을 발급하는 방식이었던 것으로 기억합니다. 둘 다 세부 옵션(토큰 권한 범위, 만료 설정 등)이 다르니 문서를 꼭 확인해야겠다고 느꼈습니다.
- "토큰은 read-only가 가능한가?" 같은 질문에서, npm은 read-only 토큰 옵션이 있고 PyPI는 프로젝트 업로드 권한 범위 토큰을 만드는 식이라 동일한 개념이지만 구현 방식이 다릅니다. 이 때문에 실무에서 어떤 토큰을 써야 하는지 결정할 때 권한 모델을 비교해보는 과정이 필요했습니다.
- 2FA의 강제 적용 범위(예: 개인 계정 vs 조직/패키지 배포 시 요구)는 레지스트리 설정이나 조직 정책에 따라 달라서, 조직 내부 정책과 레지스트리(기업 계정) 설정을 함께 확인해야 한다는 점을 놓치기 쉬웠습니다.

기본 개념 정리 (간단히)

- 2FA: 계정 보안을 높이기 위한 추가 인증. TOTP(Authenticator 앱) 방식이 일반적입니다.
- 퍼스널 액세스 토큰(PAT) / API 토큰: 비밀번호 대신 CI나 스크립트에서 사용하는 인증 수단. 범위를 제한하고 만료를 두는 것이 좋습니다.
- 최소 권한 원칙: 토큰은 필요한 권한만 부여. 예: 배포 전용 토큰, 읽기 전용 토큰 등.
- 토큰 회전: 정기적으로 토큰을 폐기하고 재발급하여 유출 리스크를 줄이는 관행.

실무에서는 이렇게 확인하면 좋겠다 — 점검 절차와 명령어 예시
아래는 제가 실무용으로 정리해 둔 체크 목록과 명령어/설정 예시입니다. 각 명령어는 상황에 따라 권한이나 레지스트리 URL을 바꿔 사용해야 합니다. 잘못 사용하면 접근이 차단될 수 있으니 테스트 환경에서 먼저 시도해보세요.

1. npm 관련 점검

- 로컬/CI 설정(.npmrc) 확인
  - CI에서 토큰을 환경 변수로 주고 .npmrc에서 참조하는 방식이 안전합니다.
    예시 (.npmrc)

  ```
  //registry.npmjs.org/:_authToken=${NPM_TOKEN}
  ```

  - NPM_TOKEN은 CI 시크릿에 저장. 로컬에는 절대 하드코딩하지 않기.

- 토큰 목록/폐기
  - 토큰 목록 확인:
    ```
    npm token list
    ```
  - 토큰 생성(자동화용, 읽기 전용 등):
    ```
    npm token create --read-only
    ```
    (옵션은 npm 버전과 레지스트리 정책에 따라 다를 수 있어 문서를 확인하세요.)
  - 토큰 폐기:
    ```
    npm token revoke <token-id>
    ```

- 2FA 확인 및 설정
  - 계정에 2FA가 설정되어 있는지 UI(https://www.npmjs.com/)에서 확인.
  - 조직이나 패키지에 대해 2FA 요구가 가능한지(조직 정책)를 확인해 두면 좋습니다.

- 배포 프로세스 예시 (CI)
  - GitHub Actions에서:
    - Repository secrets에 NPM_TOKEN 추가
    - workflow 예시 (간략)
      {% raw %}
      ```yaml
      - name: Publish to npm
        run: npm publish --access public
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
      ```
      {% endraw %}
  - 실무 팁: publish 단계를 보호 브랜치나 수동 승인(job approval)으로 감싸면 리스크를 줄일 수 있습니다.

2. PyPI 관련 점검

- 토큰 발급/사용
  - PyPI는 프로젝트 범위 API 토큰을 발급할 수 있고, 토큰은 발급 후 웹 UI에서만 확인 가능(다운로드 불가)하므로 안전하게 보관해야 합니다.
  - twine 업로드 예시:
    - 환경변수 사용:
      ```
      export TWINE_USERNAME=__token__
      export TWINE_PASSWORD=${PYPI_TOKEN}
      twine upload dist/*
      ```
    - 또는 .pypirc 예시:
      ```
      [pypi]
      username = __token__
      password = pypi-AgEIcH...
      ```
      (.pypirc 파일은 권한을 600으로 한정)

- 토큰 회전/폐기
  - PyPI 토큰은 웹 UI에서 취소(revoke) 가능. 만료 설정을 지원하는지 확인해서 유효기간을 두는 것이 좋습니다.
  - CI에서는 PYPI_TOKEN을 시크릿으로 저장하고 주기적으로 교체하는 정책을 적용하면 안전합니다.

- 2FA 확인
  - PyPI 계정에 2FA를 설정하면 계정 탈취로 인한 토큰 발급 위험을 낮출 수 있습니다. 2FA는 계정 보안 강화용으로 권장됩니다.

3. CI/시크릿 관리와 점검 절차 (공통)

- 시크릿 노출 점검
  - Git 로그나 레포지토리에 절대 토큰·비밀번호가 커밋되지 않았는지 다음과 같이 검색:
    ```
    git grep -n "NPM_TOKEN\|PYPI_TOKEN" || true
    ```
  - 또는 도구(eg. truffleHog, git-secrets)를 사용해 정기적으로 스캔.

- 시크릿 권한/범위 점검
  - CI 시크릿에 저장한 값의 권한(누가 수정 가능한지)과 사용 범위를 문서화.
  - 토큰이 특정 패키지만 업로드하도록 스코프가 설정되어 있는지 확인.

- 토큰 회전 스케줄 예시
  - 자동화 토큰: 90일 주기 교체 권장(팀 정책에 따라 조정)
  - 긴급 폐기 절차 문서화: 유출 의심 시 토큰 폐기→새 토큰 발급→CI 시크릿 갱신→릴리즈 롤백 순서

4. 패키지 공급망 관점 점검

- 패키지 소유자(maintainer) 변경 이력 확인
  - npm: `npm owner ls <package>`로 권한을 가진 사용자 리스트를 확인해 이상한 사용자가 없는지 봅니다.
  - PyPI: 웹 UI에서 프로젝트 권한(maintainers) 확인.
- 의심스러운 패키지 업로드 감지
  - 레지스트리에서 최근 퍼블리시 기록을 확인하는 모니터링(정기 스크립트 또는 웹훅).

실제로 제가 구성해 본 간단한 예제 워크플로(요약)

- GitHub Actions:
  - 세크릿: NPM_TOKEN, PYPI_TOKEN을 각각 repo secrets에 저장
  - publish job은 다음을 만족할 때만 실행:
    - 태그 푸시(ref: refs/tags/\*)
    - 특정 승인 필요(환경 approval) 또는 main 브랜치의 보호 규칙 충족
  - 로그에는 절대 시크릿이 출력되지 않도록 `set -x` 비활성화, echo에 변수 출력 금지

주의할 점(제가 실무에서 주의하려고 메모한 것)

- 문서화가 부실하면 누가 언제 토큰을 만들고 폐기했는지 추적하기 어렵습니다. 토큰 발급/폐기 로그와 담당자 기록을 남겨두면 좋습니다.
- 토큰을 브랜치나 PR 로그에 실수로 노출하면 즉시 폐기해야 합니다.
- 레지스트리 API가 업데이트될 수 있으니 CLI 옵션이나 명령은 정기적으로 문서에서 확인하는 습관을 권장합니다.

공부하면서 알게 된 추가 팁

- 최소 권한: 가능한 경우 "배포 전용" 토큰을 만들고 읽기/쓰기 권한을 나누면 더 안전합니다.
- 조직 정책: GitHub Organization, npm Organization 같은 곳에서 2FA 강제화 옵션이 있으면 팀 차원에서 켜두는 것이 좋을 것 같습니다.
- 감사 로그: npm과 PyPI 모두 일부 로그를 제공하므로(특히 기업용 계정) 정기적으로 확인하면 의심스러운 활동 탐지에 도움이 됩니다.

실무 체크리스트

- 계정 보호
  - [ ] 개인/팀 계정에 2FA 활성화 여부 확인
  - [ ] 2FA 복구 코드 안전 보관 방법 문서화
- 토큰 관리
  - [ ] CI에 사용 중인 토큰 목록 확보
  - [ ] 토큰별 권한(스코프)과 사용 목적 문서화
  - [ ] 토큰 회전 주기 설정(예: 90일) 및 자동화 여부 검토
  - [ ] 불필요한 레거시 토큰 폐기
- 레포/CI 점검
  - [ ] 레포지토리에 하드코딩된 토큰/비밀번호가 없는지 검사
  - [ ] CI 로그에 시크릿 노출 방지를 위한 설정 확인
  - [ ] 시크릿 접근 권한(누가 편집 가능한지) 최소화
- 배포 프로세스
  - [ ] 배포 작업은 보호 브랜치/태그와 연동
  - [ ] 수동 승인이 필요한 경우 승인 절차 문서화
- 모니터링·감사
  - [ ] npm/PyPI에서 배포 이력, 토큰 사용 로그(가능한 경우) 정기 확인
  - [ ] 패키지 소유자 변경, 신규 퍼블리시 알림 설정 검토
- 사고 대응
  - [ ] 토큰 유출 시 폐기·재발급·CI 갱신 절차 문서화
  - [ ] 패키지 롤백/응급 배포 차단 절차 준비

마무리하면서
이 글은 제가 공부하면서 정리한 초안이며, 레지스트리의 기능이나 CLI 옵션은 버전에 따라 바뀔 수 있습니다. 실무에 적용할 때는 반드시 각 레지스트리의 최신 문서와 조직 정책을 참고하시고, 작은 테스트로 먼저 검증해보는 것을 권합니다. 패키지 공급망 보안은 한 번에 완벽해지기보다 여러 작은 개선(2FA, 최소 권한, 토큰 회전, CI 시크릿 관리)을 쌓아가는 접근이 현실적이라고 느꼈습니다.
