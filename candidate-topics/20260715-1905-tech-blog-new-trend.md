- 검색 의도: Sigstore·in-toto·keyless signing을 기존 CI/CD에 어떻게 통합하고 기존 서명 방식과 어떤 차이가 있는지 알고 싶다.
- 실무 가치: 빌드 아티팩트 자동 서명·검증 파이프라인 구성, 키 관리를 줄이는 keyless 흐름과 rollback/검증 절차 등 바로 적용 가능한 단계별 가이드를 제공합니다.
- 최신성 포인트: 여러 오픈소스 프로젝트와 클라우드 공급자가 Sigstore 연동을 빠르게 지원하면서 관련 도구와 권고가 최근에 업데이트되고 있습니다.

### 2. 쿠버네티스 네트워킹: iptables에서 eBPF CNI로 전환할 때 체크리스트

- slug: `k8s-iptables-to-ebpf-cni-migration-checklist`
- 추천 이유: eBPF 기반 CNI가 성능·관측 측면에서 장점을 제공하면서 프로덕션 전환 사례가 늘어 운영 영향과 호환성 점검이 필요합니다.
- 검색 의도: eBPF CNI(예: Cilium 등)가 기존 iptables 기반 네트워크와 어떻게 다르고 마이그레이션 시 무엇을 확인해야 하는지 알고 싶다.
- 실무 가치: 네트워크 정책 매핑, 모니터링/디버깅 도구 변경, 롤백 플랜과 노드/플랫폼별 제한 사항 등 실제 마이그레이션에서 바로 쓰는 체크리스트를 제공합니다.
- 최신성 포인트: CNI 프로젝트들의 eBPF 기능과 권장 설정이 최근에 활발히 발전해 클러스터 네트워킹 정책을 재검토할 시점입니다.
- 확인 권장 출처: 벤더 블로그, 공식 문서, 릴리즈 노트, 커뮤니티 가이드
- 썸네일: eBPF CNI 전환 체크리스트 / 네트워크 정책·관측·디버깅이 바뀌는 지점을 중심으로 점검

### 3. 서버리스와 경량 실행환경: WASM(WASI) 런타임 도입의 운영적 고려사항

- slug: `wasm-wasi-serverless-operational-considerations`
- 추천 이유: WASM 기반 런타임이 보안·성능·시작 시간에서 장점을 제공하며 서버리스·엣지 워크로드 대체로 주목받고 있어 운영 관점의 점검이 필요합니다.
- 검색 의도: WASM/WASI 런타임을 기존 컨테이너 기반 서버리스와 비교해 비용·보안·배포·디버깅 관점에서 어떤 차이가 있는지 알고 싶다.
- 실무 가치: 런타임 선택 기준, 빌드·배포 파이프라인 변환, 관측·디버깅 전략 및 리소스 제약 대응 등 실무에서 바로 적용 가능한 체크포인트를 제공합니다.
- 최신성 포인트: 여러 클라우드 제공자와 오픈소스 런타임의 WASM 지원이 확대되며 관련 운영 가이드와 도구 연동이 최근 빠르게 갱신되고 있습니다.
- 확인 권장 출처: 공식 문서, 벤더 블로그, 릴리즈 노트, 표준 문서
- 썸네일: WASM 서버리스 운영 가이드 / 컨테이너 대비 배포·관측·보안 패턴의 차이와 적용 포인트

### 4. Kubernetes Pod 보안 모델 변화: PodSecurityAdmission(PSA) 전환 영향과 대응

- slug: `k8s-podsecurityadmission-psa-migration-impact`
- 추천 이유: 클러스터 기본 보안 모델 변화로 네임스페이스·권한 정책·배포 파이프라인을 재검토해야 할 필요가 커졌습니다.
- 검색 의도: PodSecurityAdmission으로 전환할 때 기존 PodSecurityPolicy 기반 설정을 어떻게 매핑하고 테스트해야 하는지 알고 싶다.
- 실무 가치: 정책 매핑 방법, 미리보기/감시 모드로 검증하는 절차, CI/CD에서의 자동화 적용 등 운영에서 당장 적용 가능한 대응 방안을 제공합니다.
- 최신성 포인트: Kubernetes 보안 정책 권장 사항과 기본 동작이 최근 정립되면서 클러스터 보안 설정을 시급히 점검해야 합니다.
- 확인 권장 출처: 공식 문서, 릴리즈 노트, 벤더 블로그, 커뮤니티 가이드
- 썸네일: PSA 전환 영향 분석 / 기존 PSP 정책을 PSA로 마이그레이션할 때 점검해야 할 항목

### 5. 클라우드 워크로드 신원: OIDC/Workload Identity로 시크릿 관리 전환 체크포인트

- slug: `workload-identity-oidc-secretless-transition-checkpoints`
- 추천 이유: 하드코딩된 시크릿 사용을 줄이고 OIDC 기반 워크로드 신원으로 전환하는 조직이 늘어나며 전환 영향 점검이 필요합니다.
- 검색 의도: Workload Identity나 OIDC 연동으로 서비스 계정과 시크릿 관리를 어떻게 안전하게 교체하고 로컬/CI 환경을 연결하는지 알고 싶다.
- 실무 가치: IAM 권한 매핑, 토큰 수명·갱신 관리, 로컬 개발·CI 연동 전략과 최소 권한 적용 등 실무 전환에서 필요한 체크리스트를 제공합니다.
- 최신성 포인트: 클라우드 벤더들이 워크로드 신원 기능을 강화하고 관련 권장 패턴을 업데이트하면서 보안 설정을 재점검할 필요가 커졌습니다.
- 확인 권장 출처: 공식 문서, 벤더 블로그, 보안 권고, 표준 문서
- 썸네일: 시크릿리스 전환 체크포인트 / Workload Identity 도입으로 시크릿·IAM을 안전하게 바꾸는 단계
