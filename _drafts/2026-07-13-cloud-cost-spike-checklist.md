오늘은 "클라우드 비용이 갑자기 늘었을 때" 제가 초보 개발자로서 공부하면서 정리한 점들을 차근차근 적어보려 합니다. 혼자 공부하면서 시행착오를 겪은 경험을 바탕으로, 실무에서 먼저 확인하면 좋을 항목들을 우선순위로 정리했고, 각 항목별로 바로 써볼 수 있는 명령어나 콘솔 확인 포인트도 함께 적었습니다. 전문가가 단정적으로 가르치는 글은 아니고, 제가 알게 된 것들을 조심스럽게 정리한 것이라고 이해해 주세요.

공부하면서 알게 된 점
- 클라우드 비용 증감은 단일 원인이 아니라 여러 요소가 복합적으로 작용하는 경우가 많았습니다. 예를 들어 EC2 인스턴스 증가 + S3 로그 보존 기간 연장 + 모니터링 지표의 세부 해상도 변경이 동시에 비용을 올릴 수 있습니다.
- 로그·모니터링·데이터 전송(egress) 같은 항목은 종종 비용의 큰 부분을 차지하지만 눈에 잘 안 보이는 편이었습니다. 처음에는 인스턴스 크기만 의심했는데, 로그와 스냅샷이 더 큰 요인이었던 경험이 있습니다.
- 비용 탐색 도구(예: AWS Cost Explorer, GCP Billing)에서 서비스별, 태그별로 비용을 그룹핑하는 습관이 문제를 빨리 찾는 데 도움이 됐습니다.

처음에는 헷갈렸던 부분
- "할당/사용 시간" 기반 요금과 "리저브드/스팟/스팸" 같은 예약 타입의 차이. 예컨대 Reserved Instance와 Savings Plan이 실제로 어떤 자원에 적용되는지 이해하는 데 시간이 걸렸습니다.
- 로그/모니터링 비용은 항목별(로그 이벤트 수, 로그 보관 기간, 커스텀 메트릭 수)로 과금되는데, 이걸 놓치기 쉽습니다.
- 클라우드 공급자 콘솔에 나오는 비용과 실제 청구서의 시차(알려진 지연) 때문에 "지금 당장 발생한 비용"을 파악하는 게 어렵더군요. 그래서 실무에서는 예측치와 청구서 확인을 함께 해야 한다고 느꼈습니다.

빠른 우선순위 점검 흐름 (제가 실무에서 먼저 해보는 순서)
1. 비용 대시보드(최근 일별/시간별 트렌드) 확인
2. 어떤 서비스가 비용을 많이 쓰는지 상위 서비스 확인
3. 태그/프로젝트별 비용 확인(가능하면)
4. 갑작스런 자원 증가(인스턴스, 컨테이너, 디스크, 스냅샷 등) 확인
5. 로그/모니터링/데이터 전송(egress) 설정 확인
6. 백업/스냅샷 정책과 보존 기간 확인
7. 자동 스케일 설정이나 배치 작업(크론, 데이터 파이프라인) 확인
8. 외부(마켓플레이스, 서드파티) 비용 확인

구체적인 체크 항목 및 실무용 명령 예시
아래는 AWS 중심의 예시 명령들을 포함하지만, GCP나 Azure 사용 환경에도 거의 동일한 흐름(청구 대시보드 → 리소스 인벤토리 → 로그/모니터링 → 네트워크/백업)으로 적용할 수 있습니다.

1) 비용 개요(빠른 트렌드 파악)
- 콘솔: AWS Cost Explorer 혹은 GCP Billing → 최근 7/30일 추이 확인
- AWS CLI 예시 (일별 UnblendedCost 집계)
  ```
  aws ce get-cost-and-usage \
    --time-period Start=2026-07-01,End=2026-07-13 \
    --granularity DAILY \
    --metrics "UnblendedCost" \
    --group-by Type=DIMENSION,Key=SERVICE
  ```
  이 결과로 어느 서비스가 갑자기 늘었는지 1차 파악합니다. (참고: Cost Explorer API는 별도 권한이 필요합니다.)

2) 상위 비용 발생 서비스/태그 확인
- 콘솔에서 "Group by: service, tag"로 나눠보면 빠릅니다.
- 태그가 없다면 비용 책임자 추적이 어려워서, 실무상 태깅 정책을 빨리 도입하는 편이 좋습니다.

3) 컴퓨트(가상머신/컨테이너) 점검
- EC2 인스턴스 목록과 상태:
  ```
  aws ec2 describe-instances \
    --query 'Reservations[*].Instances[*].[InstanceId,InstanceType,State.Name,Tags]' \
    --output table
  ```
- 과다한 인스턴스 수나 큰 타입(a1.4xlarge 같은)이 갑자기 늘었는지 확인합니다.
- 오토스케일링(ASG) 이벤트 확인: 콘솔의 Auto Scaling Activities 또는 CloudWatch 이벤트 확인.
- Kubernetes(예: EKS, GKE) 점검:
  ```
  kubectl get nodes -o wide
  kubectl top nodes
  kubectl top pods --all-namespaces
  kubectl get hpa --all-namespaces
  ```
  HPA(수평 자동스케일)가 원치 않게 빠르게 스케일 업 되진 않았는지 확인합니다.

4) 스토리지와 스냅샷(백업) 점검
- S3(오브젝트 스토리지): 버킷 수와 보관 클래스, 수명주기 설정 확인
  ```
  aws s3 ls
  aws s3api get-bucket-lifecycle-configuration --bucket my-bucket
  ```
  특히 로그/접근 기록이 S3에 많이 쌓이는 경우, 보관 클래스(Glacier 등)나 수명주기를 통해 비용을 낮출 수 있습니다.
- EBS 스냅샷 누적 확인:
  ```
  aws ec2 describe-snapshots --owner-ids self \
    --query 'Snapshots[*].[SnapshotId,VolumeId,StartTime,State,VolumeSize]' \
    --output table
  ```
  오래된 스냅샷이 쌓여서 비용을 올리지 않는지 확인합니다. 스냅샷은 특히 대용량에서 비용 비중이 큽니다.

5) 로그·모니터링 비용 점검
- CloudWatch Logs(또는 GCP Stackdriver, Azure Monitor) 보관 기간과 로그 이벤트 수 확인
  ```
  aws logs describe-log-groups --query 'logGroups[*].[logGroupName,retentionInDays]' --output table
  journalctl --disk-usage          # 로컬 시스템 로그 사용량 확인 (Linux)
  ```
  - 커스텀 메트릭 수가 많거나, 고해상도(초 단위) 수집을 하고 있진 않은지 확인합니다.
  - 로그 보존기간을 줄이거나, 로그 필터링/샘플링을 적용하는 것이 비용에 도움이 됩니다.

6) 네트워크(데이터 전송) 확인
- 데이터 전송(인터넷 egress)은 의외로 큰 비용 원인입니다. S3 → 인터넷, 서비스 간 대량 데이터 전송, 외부 백업 등이 있는지 확인하세요.
- VPC Flow Logs 같은 추가 로깅도 비용을 유발할 수 있습니다.
- NAT Gateway는 시간당 + 데이터 전송 비용이 있으니, NAT Gateway 사용량을 점검합니다.

7) 오토메이션/배치 작업 확인
- 크론이나 CI 파이프라인, 데이터 파이프라인이 예상 외로 자주 실행되지는 않는지 확인합니다. 예: 테스트 환경에서 매시간 대량의 배치가 돌면 비용이 커집니다.
- Jenkins/GitHub Actions 등에서 대규모 워크플로우가 반복 수행되는지 점검하세요.

8) 오케스트레이션·컨테이너 이미지·Docker 점검
- Docker 환경(로컬 혹은 자가 호스팅 레지스트리)에서 이미지/컨테이너가 많이 쌓였는지 확인:
  ```
  docker system df
  docker images
  docker ps -a
  ```
  이미지 보존/청소 정책이 없으면 디스크, 레지스트리 스토리지가 늘어납니다.

9) 오케스트레이션 수준의 리소스 누수(예: 오래된 Load Balancer, EIP)
- 사용하지 않는 Elastic IP, Load Balancer, NAT Gateway, RDS 읽기 복제본 등이 비용 원인이 되는 경우가 많습니다. 리소스 인벤토리를 통해 "사용 중인지" 확인합니다.

10) 써드파티/마켓플레이스 요금 확인
- 마켓플레이스에서 구독한 소프트웨어가 비용을 유발할 수도 있습니다. 청구 항목 상세를 확인하세요.

문제 원인 파악 시 사용하면 좋은 추가 도구들
- 비용 알림/예산(Budgets) 설정: Budget을 만들어서 초과 시 알람 받기
- Cost Allocation Tags: 팀/프로젝트 단위 비용 분리
- Cost Anomaly Detection: 이상 비용 탐지(Cloud provider 제공 기능)
- 쿼리 기반 리포트: Cost Explorer 쿼리·그룹핑을 통해 문제 원인 세분화

실무에서는 이렇게 확인하면 좋겠다 (권장 절차)
1. 시간 범위 좁혀서(증가 직전 ~ 현재) Cost Explorer로 서비스별 변화 확인
2. 상위 5개 서비스와 상위 10개 리소스(태그 기준)를 우선 조사
3. 컴퓨트(인스턴스/오토스케일링), 스토리지(스냅샷/S3), 네트워크(egress), 모니터링(로그/메트릭) 각각의 최근 변경 이력을 확인
4. 배포/크론/데이터 파이프라인 로그를 살펴 예기치 않은 반복 작업이 있는지 확인
5. 의심되는 리소스는 일시 중지(스냅샷/백업 후)하거나 태그로 분리하여 비용 변화 추적
6. 문제 원인 파악 후에는 보존 기간, 수명주기, 오토스케일 정책, 예약 인스턴스/스팟 활용을 검토

주의사항(제가 배운 것)
- "바로 삭제"하기 전에 사용 여부와 종속성을 반드시 확인하세요. 특히 데이터베이스나 스냅샷, 디스크는 복구 불능 상태로 만들 수 있습니다.
- 비용 감소를 위해 과도하게 리소스를 축소하면 서비스 가용성이나 성능에 영향을 줄 수 있으니, 안전한 테스트·스케줄을 거치세요.
- 클라우드 공급자의 비용 계산은 복잡하고 리전/요금제/할인 적용 여부에 따라 달라집니다. 확실하지 않으면 청구서 샘플을 담당자와 함께 확인해 보세요.

간단한 조사용 스크립트 예제 (AWS 중심)
- 최근 30일간 상위 서비스 비용 요약(간단)
  ```
  aws ce get-cost-and-usage \
    --time-period Start=$(date -d '30 days ago' +%F),End=$(date +%F) \
    --granularity MONTHLY \
    --metrics "UnblendedCost" \
    --group-by Type=DIMENSION,Key=SERVICE
  ```
- EC2 인스턴스와 태그 정보(테이블 출력)
  ```
  aws ec2 describe-instances \
    --query 'Reservations[*].Instances[*].[InstanceId,InstanceType,State.Name,Placement.AvailabilityZone,Tags]' \
    --output table
  ```
- 로그 그룹별 보존기간 확인
  ```
  aws logs describe-log-groups --query 'logGroups[*].[logGroupName,retentionInDays]' --output table
  ```

공부하면서 알게 된 작은 팁들
- 태그 정책을 빨리 도입하면 비용 원인 추적이 훨씬 쉬워집니다. 실무에선 "project", "env", "owner" 같은 태그를 필수로 적용해 두는 편이 안전합니다.
- 로그/메트릭은 "필요한 것만" 수집하되, 필요하다면 Sampling/Retention을 조절해서 비용을 통제하는 것이 실무에서 효과적입니다.
- 장기적 관점에서 예약 인스턴스/절약 플랜을 도입하면 큰 비용 절감이 가능하지만, 우선은 사용량 패턴을 충분히 분석한 후에 결정하는 것이 좋습니다.

마무리하면서 (조심스러운 어투)
제가 정리한 내용은 초보자가 실무에서 빠르게 원인을 좁히고 기본 조치를 취하는 데 초점을 둔 체크리스트입니다. 환경마다 과금 구조와 설정이 다르기 때문에, 이 글의 방법들이 항상 정답은 아닐 수 있습니다. 다만 처음 이 문제를 마주했을 때 당황하지 않고 순서대로 점검하는 데는 도움이 될 거라고 생각합니다. 혹시 더 구체적인 클라우드 제공사(AWS/GCP/Azure)나 환경(EKS, RDS, GCS 등)에 대해 알고 싶으시면 그에 맞춘 점검 절차를 이어서 정리해 보겠습니다.

## 관련 이미지 주제
1. 클라우드 비용 대시보드와 서비스별 비용 막대 그래프가 표시된 단순한 일러스트
2. 리소스 인벤토리(서버, 스토리지, 데이터베이스)를 체크하는 체크리스트 아이콘 일러스트

실무 체크리스트
- [ ] 비용 대시보드에서 최근 일별/주별/월별 추이 확인
- [ ] 서비스별 상위 비용 항목 5개 파악
- [ ] 태그/프로젝트별 비용 분배 확인(태그 없으면 태깅 정책 도입 검토)
- [ ] 인스턴스/오토스케일링/컨테이너 스케일 상태 점검 (kubectl, aws ec2 등)
- [ ] 스토리지(S3/GCS)와 스냅샷 누적 여부 확인 및 수명주기 설정 검토
- [ ] 로그·모니터링 보존 기간과 커스텀 메트릭 수 확인
- [ ] 데이터 전송(egress) 패턴과 NAT/LoadBalancer 사용량 확인
- [ ] 배치 작업/크론/CI 파이프라인의 빈도와 최근 변경점 확인
- [ ] 불필요한 공인 IP, 미사용 로드밸런서, 오래된 스냅샷 등 오키퍼 리소스 제거(삭제 전 의존성 확인)
- [ ] 예산(Budgets)과 알람, 비용 이상탐지 설정 적용

읽어주셔서 감사합니다. 다음에는 특정 클라우드(AWS/GCP/Azure)별로 보다 상세한 점검 명령과 콘솔 위치를 정리해 보려고 합니다.
